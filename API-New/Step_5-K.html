Dual-Lane Architecture in OpenAPI Paths

The TML framework expresses its foundational Binary-Ternary Parallel Architecture through deliberate separation of OpenAPI path structures, creating two distinct but interoperable API surfaces. This architectural pattern embodies the principle that the binary inference engine and ternary governance coprocessor operate as sovereign peers, with the binary system proposing actions and the ternary system exercising absolute veto authority.

Permission Token and No Log = No Action Enforcement

The No Log = No Action principle is enforced at the schema level through conditional field requirements and cross-schema validation. This creates a structural impossibility: a +1 state cannot exist without a valid PermissionToken, which in turn requires an anchored MoralTraceLog.

Schema Enforcement Chain

1
StateEnvelope Conditional Requirement

If current_state: 1, then permission_token is mandatory

2
PermissionToken Schema Validation

Token must contain log_hash, signer_key_id, and valid signature

3
MoralTraceLog Existence

log_hash must correspond to an anchored log entry

PermissionToken Schema Structure

Required Fields
  • log_hash - SHA-256 of anchored log
  • epoch_timestamp - Unix epoch milliseconds
  • signer_key_id - HSM key identifier
  • lane_origin - Must be "ANCHORING"
Cryptographic Binding
  • signature_value - ECDSA signature
  • merkle_root - Inclusion proof anchor
  • signature_algorithm - Algorithm agility

Deliverable G: future_blocked_appendix.md

A catalog of aspirational features blocked by implementation gaps from Monograph Section 10, with specific mitigations noted and FUTURE status assigned.

Real-Time Per-Token Blockchain Anchoring

Architectural Target

Every Permission Token individually and irreversibly recorded on public blockchain at issuance, creating instantaneous global visibility into system moral decisions.

Status: FUTURE

Blocked by Section 10.2: Latency-Legitimacy Dilemma, Throughput Asymmetry

Throughput Asymmetry Analysis

Component Throughput
Global AI Inference ~10⁶ TPS
Ethereum Mainnet ~15 TPS
ZK-Rollup (Theory) ~2,000 TPS

Mitigation: Merkle-Batched Anchoring

Current specification implements Merkle-batched anchoring through POST /blockchain/anchor and GET /audit/verifications/merkle-root/{logHash}, reducing blockchain write frequency by thousands while preserving individual token verifiability.

Quantum-Proof Signature Migration

Architectural Target

Complete migration to post-quantum cryptography (PQC) algorithms for all signing operations, protecting against quantum computing threats to current ECDSA/Ed25519 implementations.

Status: FUTURE

Blocked by Section 10.7: Quantum Computing Horizon Risks

Performance Penalties

Algorithm Sig Size Verify Time
ECDSA (secp256k1) 64 bytes ~0.1 ms
CRYSTALS-Dilithium (L2) ~2,420 bytes ~0.5 ms
CRYSTALS-Dilithium (L5) ~4,595 bytes ~1.2 ms

Mitigation: Algorithm Agility

Current specification implements algorithm agility through SignatureBlock.signatureAlgorithm enum, with reserved identifiers for NIST PQC standards. This provides migration path while maintaining current performance characteristics.

Hardware Moral Processing Units (MPUs)

Architectural Target

Dedicated hardware accelerators that perform ternary logic evaluation and moral governance decisions with hardware-enforced latency and security guarantees.

Status: FUTURE

Blocked by Section 10.5: Hardware-Constitutional Integration Complexity

Implementation Challenges

  • No commercial hardware currently implements ternary logic at scale
  • Security verification of hardware moral circuits remains unsolved
  • Supply chain integrity for constitutional hardware components
  • Update mechanisms for evolving moral frameworks

Mitigation: Software-First Implementation

Current specification implements all moral governance logic in software with HSM-backed cryptography, providing functional equivalent while awaiting hardware maturity.

Cross-Jurisdiction Custodian Quorum in <500ms

Architectural Target

Achieve global unanimous quorum across all six custodians in under 500ms, enabling truly distributed governance with minimal latency impact.

Status: FUTURE

Blocked by Section 10.4: Geographical Distribution Latency Constraints

Physical Constraints

Route Distance Min Latency
London ↔ Singapore ~10,800 km ~72 ms
New York ↔ Tokyo ~10,900 km ~73 ms
Global Round Trip ~40,000 km ~267 ms

Mitigation: Regional Quorum with Fallback

Current specification implements regional quorum with cross-jurisdiction fallback through GET /regulator/custodian-quorum-status, providing best-effort global coordination while maintaining anchoring lane latency targets.

Immutable Ledger with Native GDPR Article 17 Compliance

Architectural Target

Provide complete immutability for moral governance records while simultaneously supporting the "right to be forgotten" through cryptographic erasure mechanisms.

Status: FUTURE

Blocked by Section 10.8: Erasure Paradox

The Erasure Paradox

The Conflict

GDPR Article 17 requires data erasure upon request, while TML's No Log = No Action principle requires permanent, immutable records for every actuation.

Current Gap

No known cryptographic technique can simultaneously provide:

  • • Complete immutability for governance
  • • Complete erasure for privacy rights
  • • Verifiable compliance with both requirements

Mitigation: Pseudonymization with Selective Disclosure

Current specification implements pseudonymization and selective disclosure through anonymization_level parameter in POST /regulator/bulk-evidence-export, providing privacy protection while maintaining log integrity. However, this does not satisfy full Article 17 requirements.
