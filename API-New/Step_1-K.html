Executive Summary

The TML Specification Architecture v3.3 defines a comprehensive OpenAPI 3.1.0 API contract for the Ternary Moral Logic (TML) framework, implementing a dual-lane architecture with strict separation between binary inference (proposing actions) and ternary governance (authorizing execution via cryptographically signed Permission Tokens).

20+ Endpoints

Comprehensive coverage across Inference, Anchoring, Escalation, Refusal, Emergency Override, Auditor, Regulator, Redress, and Gateway lanes.

Tiered Security

mTLS/ServiceAccountJWT for Inference; HSM-SignedJWT/MutualTLS for Anchoring; CA-VettedJWT/IPAllowlist for oversight.

Schema Enforcement

"No Log = No Action" principle enforced through conditional validation requiring Permission Tokens for State +1.

The specification implements the Eight Pillars framework with mandatory x-tml-pillar, x-tml-monograph-ref, and x-tml-implementation-status annotations on all endpoints, providing comprehensive regulatory mapping and implementation guidance.

Security Architecture

The TML security architecture implements a tiered authentication and authorization model that maps directly to the Dual-Lane Latency Architecture's separation of concerns. Six distinct security schemes are organized into three logical tiers that reflect the sensitivity and consequence of operations in each lane.

Scheme Identifier Type Lane Assignment Security Tier Key Characteristic
mTLS mutualTLS Inference Lane Transport Bidirectional certificate authentication; <2ms latency target
ServiceAccountJWT http (bearer) Inference Lane Application Service account tokens; machine-to-machine identity
HSM_SignedJWT http (bearer) Anchoring/Governance Application Hardware Security Module signing; non-repudiation
MutualTLS mutualTLS Anchoring/Governance Transport Independent PKI hierarchy from Inference Lane
CA_VettedJWT http (bearer) Auditor/Regulator/Redress Application Certificate Authority identity vetting; human principals
IPAllowlist apiKey (header) Auditor/Regulator/Redress Network Source IP validation; defense-in-depth for sensitive operations

Implementation Status Analysis

The TML Specification Architecture employs a rigorous implementation status classification system that acknowledges both current capabilities and future constraints. This transparency enables stakeholders to make informed decisions about deployment readiness and regulatory compliance.

Implementation Status Classification

SHIPPING

Buildable with 2025 production libraries and infrastructure; no known blockers from Section 10.

Examples: /decisions, /audit/merkle-root, /gateway/route

BETA

Buildable with documented tradeoffs; latency, cost, or throughput penalties acceptable for regulated deployments.

Examples: /anchoring/logs, /escalations, /redress/challenge

FUTURE

Blocked by a named constraint from Section 10 (Implementation Gap); requires architectural evolution.

Examples: /emergency/shutdown, /regulator/evidence/export

Implementation Gap Analysis

Real-time Per-Token Blockchain Anchoring

Throughput asymmetry between AI decision rates and blockchain transaction rates creates an unsustainable cost and latency model.

Constraint: Section 10.1 - Blockchain throughput limitations

Quantum-Proof Signature Migration

Post-quantum cryptographic algorithms are not yet standardized or widely supported in HSM infrastructure.

Constraint: Section 10.3 - PQC algorithm standardization

Hardware Moral Processing Units (MPUs)

Specialized hardware for ternary logic and pillar enforcement not commercially available at required scale.

Constraint: Section 10.4 - Hardware availability and cost

Cross-Jurisdiction Custodian Quorum

Maintaining <500ms latency for six-custodian distributed consensus across global jurisdictions presents network reliability challenges.

Constraint: Section 10.6 - Geographic latency constraints

Immutable Ledger with GDPR Article 17 Compliance

The Erasure Paradox - reconciling immutable blockchain records with the right to be forgotten presents fundamental architectural conflicts.

Constraint: Section 10.7 - Privacy vs. immutability conflict

Mitigation Strategies

Merkle-Batched Anchoring

Batch processing of Permission Tokens with periodic Merkle root anchoring rather than per-token blockchain transactions.

Status: SHIPPING - Implemented in /audit/merkle-root

Cryptographic Erasure

Zero-knowledge proofs and cryptographic accumulators to prove removal without altering blockchain history.

Status: BETA - Research implementations available

Hybrid Custodian Model

Strategic custodian placement in major jurisdictions with redundant network paths to maintain quorum reliability.

Status: BETA - Test deployments operational

Gradual PQC Migration

Hybrid signature schemes combining classical and post-quantum algorithms for transition period compatibility.

Status: BETA - Standardization in progress

Conclusion

The TML Specification Architecture v3.3 represents a comprehensive framework for implementing ethical AI governance through rigorous API design and architectural discipline. By enforcing the "No Log = No Action" principle at the schema level and implementing dual-lane architecture with cryptographic separation of concerns, the specification provides a foundation for trustworthy AI systems.

Key Achievements

  • Complete API Specification: 20+ endpoints covering inference, governance, escalation, audit, and emergency functions
  • Schema-Level Enforcement: "No Log = No Action" through conditional validation requiring Permission Tokens
  • Eight Pillars Integration: Comprehensive mapping of architectural principles to API capabilities
  • Regulatory Compliance: Explicit linkage to EU AI Act, NIST RMF, ISO 42001, and international frameworks

The specification's transparent acknowledgment of implementation gaps—particularly around quantum cryptography, cross-jurisdiction latency, and the erasure paradox—demonstrates responsible engineering practice. By classifying these as FUTURE capabilities with explicit blocking constraints, the architecture guides implementers toward achievable deployments while maintaining aspirational targets.

Future Evolution

As the Implementation Gap constraints are resolved through technological advancement and standardization, the TML Specification Architecture will evolve to incorporate:

  • • Quantum-resistant signature schemes
  • • Real-time blockchain anchoring at scale
  • • Hardware Moral Processing Units (MPUs)
  • • Enhanced cross-jurisdiction custodian coordination
  • • Privacy-preserving immutable ledgers

The success criteria for the specification—enabling engineers to build compliant systems without re-reading the monograph, while allowing auditors to verify compliance without reading implementation code—establishes a new standard for ethical AI governance documentation. This dual-audience approach ensures that the TML framework serves both technical implementation and regulatory oversight needs.
