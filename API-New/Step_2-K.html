Introduction

The tml_schema.json deliverable represents the canonical data contract for the Ternary Moral Logic (TML) framework. As specified in the TML Constitutionalization Monograph, this schema bundle defines the complete set of JSON Schema Draft 2020-12 definitions that govern moral decision logging, permission token issuance, and cryptographic enforcement across the dual-lane architecture.

Schema Bundle Scope

This document encompasses all schema definitions referenced by the OpenAPI specification, including State Envelope, TSLF variants, Permission Token, and Goukassian Promise artifacts, with strict adherence to the Eight Pillars architecture.

$defs Organization

Centralized definitions with strict unevaluatedProperties: false enforcement, ensuring closed-content validation and preventing schema confusion attacks.

Discriminator Pattern

oneOf with discriminator-based polymorphism for TSLF variants, enabling efficient single-pass validation compatible with <500ms anchoring targets.

No Log = No Action

Conditional Permission Token requirements enforced at schema level, ensuring cryptographic decoupling of inference from actuation.

Moral Trace Log Format (TSLF) Variants

Discriminator-Based Polymorphism

The TSLF schema employs oneOf with discriminator to achieve type-safe polymorphism with O(1) validation performance. This pattern enables the Anchoring Lane's <500ms latency target by avoiding exhaustive trial validation of complex forensic schemas.

Discriminator Configuration

{
  "discriminator": {
    "propertyName": "tmlState",
    "mapping": {
      "0": "#/$defs/TSLFState0",
      "1": "#/$defs/TSLFStatePlus1",
      "-1": "#/$defs/TSLFStateMinus1"
    }
  }
}

Permission Token Schema

Cryptographic Enforcement of No Log = No Action

The Permission Token is the schema-level manifestation of TML's core principle. Its presence in a Proceed state envelope provides cryptographic proof that the Anchoring Lane has independently verified moral permissibility before actuation.

Required Fields

logHash

SHA-256 digest of the anchored Moral Trace Log, creating immutable binding between authorization and forensic record. 

"pattern": "^[a-fA-F0-9]{64}$"

epochTimestamp

Qualified timestamp from Anchoring Lane, ensuring temporal authority and preventing replay attacks. 

"type": "integer",
"minimum": 1577836800000

signerKeyId

HSM-backed key identifier of the signing custodian, enabling key rotation and compromise recovery. 

"format": "ANCHORING-HSM-{CUSTODIAN}-{SEQ}"

Security Constraints

laneOrigin

Source lane enum enforcing cryptographic decoupling. Must be ANCHORING for valid Proceed authorization. 

"enum": ["INFERENCE", "ANCHORING"]

anchoringSignature

ECDSA-SHA256 signature over complete token, verifiable against signerKeyId for non-repudiation. 

"pattern": "^[a-fA-F0-9]{128}$"

merkleRootVerification

Batch Merkle root reference and inclusion proof, acknowledging Implementation Gap with epoch-scoped batching. 

"verificationStatus": 
  ["PENDING", "COMPLETE"]

Lane Origin Enforcement

The laneOrigin constraint implements the fundamental separation between binary Inference Lane proposals and ternary Anchoring Lane authorizations. Any token with laneOrigin: INFERENCE in a Proceed envelope constitutes a schema violation, reflecting the architectural requirement that "the binary system proposes; the ternary system dictates."

Inference Lane: Proposes Actions
Anchoring Lane: Authorizes Execution
